Pipeline
01
rate-limited client
utils/api_client.py wraps the API with a 5 req/s budget. Every call sleeps to the next slot in the window. No bursts, no 429s in practice.
02
fetch top 200
GET /locations/global/rankings/players returns the leaderboard. We keep the player tags only — everything else is requested per-player.
03
per-player deck pull
200 sequential GET /players/{tag} calls, each returning the player's current deck of 8 cards. With the 5 req/s budget this takes ~40 seconds — the bottleneck.
04
aggregate card usage
card_analyzer.py walks the 200 decks, increments a counter per card, divides by 200. Returns the list sorted by frequency descending.
05
battle replay
For a single player tag the user enters, GET /players/{tag}/battlelog returns the last 25 battles. battle_analyzer.py extracts crowns, trophy delta, opponent deck, win/loss.
06
bot detector
bot_detector.py runs ten weighted signals against a profile + battlelog. Returns score, verdict, and each signal's fired/clean state. Anti-mystery: no model, no opacity.
07
opponent sweep
RecentOpponentScanner walks your last 25 battles, dedupes opponent tags, scores each. ~50 API calls at 5 req/s ≈ 10s. Sorted by suspicion, descending.
08
card synergy
synergy.py builds a co-occurrence count from top-200 decks. For any card, returns the seven cards most often slotted alongside it. Cached per sweep.
09
render
Flask hands the structured JSON to Jinja, which renders five views. No SPA, no framework, no state on the server beyond the API key.
Code shape

The whole rate-limited API wrapper is small enough to read in one screen.

# clash/utils/api_client.py
class ClashRoyaleAPI:
    BASE = "https://api.clashroyale.com/v1"
    RATE = 5  # requests per second, ceiling enforced by Supercell

    def __init__(self, key):
        self.h = {"Authorization": f"Bearer {key}"}
        self.last = 0

    def _throttle(self):
        gap = 1.0 / self.RATE
        wait = gap - (time.time() - self.last)
        if wait > 0: time.sleep(wait)
        self.last = time.time()

    def get(self, path):
        self._throttle()
        url = f"{self.BASE}{path}"
        r = requests.get(url, headers=self.h, timeout=10)
        r.raise_for_status()
        return r.json()

    def top_players(self, location="global"):
        return self.get(f"/locations/{location}/rankings/players")["items"]

    def player(self, tag):
        return self.get(f"/players/%23{tag.lstrip('#')}")

The API key must be whitelisted to the IP the requests come from. For local dev that's your home IP; for any hosted version it's a fixed egress IP. This is why the live demo on this page is a UI layout rather than a real query — GitHub Pages has no backend, and a free PaaS would need its own whitelist entry.

Stack
Flask 2.0 · Werkzeug 2.0.3 pin requests + bespoke 5 req/s throttle Jinja2 templates vanilla JS for chart + forms Supercell API · official python-dotenv no ML · ten transparent heuristics
What this is not

Not a deck builder. Not a matchmaking predictor. Not an ML project — there is no model anywhere in this repo. The bot detector is ten weighted heuristics, each readable in two lines. The point was to learn how a free-tier game API holds up under a real workload (200 sequential pulls, rate-limited) and to write the smallest possible analytics surface on top of it.

Not a Supercell ToS violation. The bot detector is a "make a judgment call I'd make manually anyway" tool — it reads only public profile fields and your own battlelog. It does not auto-report, scrape between accounts, or touch any endpoint outside the official rate budget.

Not a substitute for RoyaleAPI / Stats Royale. Those services exist, do this at scale, and have IP-whitelisted infra. This repo is the proof-of-concept I built before deciding whether to graduate it into something hosted.
